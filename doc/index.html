Regras de Usuário

Três operações com lógica de negócio: cadastro com segurança, autenticação e remoção com integridade referencial.

👤 Cadastro — registerUser()

  • Verifica se e-mail já existe → DuplicateUserError
  • Encripta senha com bcrypt antes de persistir
  • Garante que a senha em texto plano nunca é salva

🔑 Autenticação — AuthenticateUser()

  • Busca usuário por e-mail → UserNotFoundError
  • Compara hash via bcrypt.compare() → WrongPasswordError
  • Retorna o usuário somente se ambas as checagens passam

🗑️ Remoção — removeUserByEmail()

  • Verifica existência → UserNotFoundError
  • Verifica aluguéis abertos → UserHasOpenRent
  • Só remove se usuário existe e não tem pendências

⭐ Proration — PricingService

  • Cobrança mínima de 1 hora, blocos de 30 min (round up)
  • Adicional +20% em fins de semana
  • Desconto −15% para aluguéis > 8 horas
  • Regras 3 e 4 aplicam-se simultaneamente
Regra Principal

Proration — Cobrança Proporcional

A regra de proration é o coração testável da aplicação. Ela combina quatro sub-regras que interagem entre si, gerando múltiplos caminhos de execução e uma superfície de teste rica para BVA, MCDC e Teste de Mutação.

1

Cobrança Mínima de 1 Hora

Qualquer aluguel, independente da duração real, é cobrado como no mínimo 1 hora inteira.

5 min alugado → 1h cobrada
59 min alugado → 1h cobrada
2

Blocos de 30 Minutos (round up)

Após a 1ª hora, qualquer fração de 30 min conta como o bloco completo.

1h 01min → 1.5h cobrada
1h 31min → 2.0h cobrada
3

Adicional de Fim de Semana (+20%)

Aluguéis iniciados no sábado ou domingo têm 20% de acréscimo sobre o total.

2h × R$10 no sábado →
R$24,00 (vs R$20 na semana)
4

Desconto Longa Duração (−15%)

Duração prorateada superior a 8 horas concede 15% de desconto.

9h × R$10 na semana →
R$76,50 (vs R$90 sem desconto)
Regras 3 e 4 aplicam-se simultaneamente. Um aluguel de 9h no sábado resulta em: 90 × 1,20 × 0,85 = R$ 91,80. Essa interação é exatamente o que o critério MCDC cobre.
src/rent/service/pricing.service.ts 
calculateRentAmount(startDate: Date, endDate: Date, valuePerHour: number): number {
  const rawHours = (endDate.getTime() - startDate.getTime()) / (1000 * 60 * 60);
  const proratedHours = this.prorateHours(rawHours);   // regras 1 e 2

  let amount = proratedHours * valuePerHour;

  if (this.isWeekend(startDate)) amount *= 1.2;   // regra 3
  if (proratedHours > 8)        amount *= 0.85;  // regra 4

  return Math.round(amount * 100) / 100;
}

prorateHours(rawHours: number): number {
  if (rawHours <= 0) return 0;
  return Math.max(1, Math.ceil(rawHours * 2) / 2);  // mín 1h, blocos de 0.5h
}

MCDC — calculateRentAmount()

Duas condições: W = isWeekend(startDate)  |  L = proratedHours > 8. Cada condição varia independentemente e altera o resultado.

Condição W (fim de semana)

+20%
Quando W muda F→T (mantendo L=F), o resultado muda de R$20 para R$24.

Condição L (> 8 horas)

−15%
Quando L muda F→T (mantendo W=F), o resultado muda de R$20 para R$76,50.

Teste de Mutação — Stryker 9

O Stryker gerou 34 mutantes modificando operadores, constantes e condições do PricingService. Exemplos de mutantes gerados e mortos:

Mutante (operação)LinhaOriginal → MutadoMorto por
EqualityOperator24 rawHours <= 0 → rawHours < 0 returns 0 for exactly 0 hours
ArithmeticOperator25 rawHours * 2 → rawHours / 2 rounds up to 1.5h for 1h01m
ConditionalExpression31 isWeekend() → false MCDC W=T, L=F
ArithmeticOperator32 amount *= 1.2 → amount *= 1 MCDC W=T, L=F
EqualityOperator35 proratedHours > 8 → proratedHours >= 8 boundary exactly 8h (no discount)
ArithmeticOperator36 amount *= 0.85 → amount *= 1 MCDC W=F, L=T
Mutante equivalente eliminado. Na primeira execução, o Stryker encontrou 1 sobrevivente: rawHours < 1 → rawHours <= 1, que produzia o mesmo resultado (ambos retornam 1h quando rawHours=1). A solução foi substituir o branch duplo por Math.max(1, Math.ceil(rawHours * 2) / 2), eliminando a ambiguidade e chegando a 100% de mutation score.
